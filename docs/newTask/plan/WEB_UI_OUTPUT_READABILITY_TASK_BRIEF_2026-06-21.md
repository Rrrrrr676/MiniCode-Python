## 1. 背景与问题

当前 Web 页面已经完成三栏独立滚动、智能跟随、统一时间线和长内容基础保护，但回答展示仍接近“原始日志”，没有形成清晰的阅读层级。

浏览器截图暴露出以下问题：

1. `#`、`###`、`**bold**`、反引号和 `---` 直接显示，Markdown 没有完整渲染；
2. 连续多个 `read_file` 工具卡占用大量纵向空间，真正回答被工具日志淹没；
3. “Back to latest”位于正文中央，会遮挡代码块和段落；
4. 长回答缺少目录、标题跳转和章节折叠，定位内容成本较高；
5. 纯文本代码块统一显示为 `text`，工具栏视觉权重偏高；
6. 达到工具步数上限时只显示普通 Assistant 文本，用户不容易理解状态和下一步操作；
7. 标题、列表、行内代码和正文没有明确的视觉层级，长篇内容扫描效率低。

本任务重点优化“结果如何被阅读”，不改变 Agent 核心推理流程。

## 2. 优化目标

任务完成后应达到：

1. MiniCode 输出中的常见 Markdown 能被安全、稳定地渲染；
2. 重复工具调用折叠为紧凑摘要，同时保留完整可追溯信息；
3. 智能跟随按钮不遮挡任何正文、代码或交互控件；
4. 长回答可以通过标题目录快速浏览和跳转；
5. 代码、普通文本、标题、列表和状态提示具有清晰层级；
6. 工具步数耗尽等非正常完成状态有明确解释和可执行的下一步；
7. 桌面端和移动端均保持可访问、无重叠、无横向页面溢出。

## 3. 必须完成范围

### 3.1 安全 Markdown 渲染

- 使用成熟 Markdown AST 渲染方案，建议 `react-markdown` 配合 GFM 插件；
- 支持标题、段落、粗体、斜体、删除线、行内代码、列表、任务列表、引用、分隔线、表格和链接；
- fenced code block 继续支持语言标识、复制、横向滚动和长代码折叠；
- 默认不执行 HTML，不启用不受控 raw HTML；
- 链接仅允许安全协议，外部链接继续设置 `target="_blank"` 和 `rel="noreferrer noopener"`；
- 流式输出中的未闭合围栏、表格或强调符号不得导致整页重排或脚本执行；
- 不使用正则拼接 HTML，不使用 `dangerouslySetInnerHTML`；
- Markdown 解析失败时回退为安全纯文本，不丢失回答内容。

### 3.2 回答排版系统

- 为 H1/H2/H3 建立明确但克制的字号、字重和上下间距；
- 正文最大阅读宽度建议控制在 760–880px，避免超长行；
- 中文正文行高建议为 1.65–1.8，英文和代码不得挤压；
- 行内代码使用低对比底色和等宽字体，不能与 fenced code block 混淆；
- 列表缩进、引用边线、表格边框和分隔线需形成稳定层级；
- 表格在窄屏下使用自身横向滚动，不撑宽 document；
- 回答中的连续空行需要收敛，避免无意义大块留白；
- 保持现有深色/浅色自动主题可读性，对比度不只依赖颜色表达状态。

### 3.3 连续工具调用聚合

- 同一 turn 内相邻、同类型的已完成工具调用聚合为一个 `ToolGroup`；
- 示例：连续 13 次 `read_file` 显示为“Read 13 files · 34ms”；
- 摘要显示调用数量、成功/失败数量、累计耗时和当前状态；
- 默认折叠，展开后按原 seq 展示每次调用、输入摘要、输出摘要和耗时；
- 任一调用失败时摘要必须明确显示失败，不得被成功调用掩盖；
- 权限请求、错误、Assistant 消息或用户消息会中断工具分组；
- 正在运行的工具保持实时可见，完成后再并入稳定分组；
- 分组不得改变 Store 中原始事件顺序和工具归属，仅改变呈现层；
- 聚合组件支持键盘展开，具有清晰的 accessible name 和状态文本。

### 3.4 “回到最新”按钮重定位

- 按钮从正文中央移到 transcript 右下角，并位于 composer 上方；
- 按钮不能覆盖正文、代码块、表格、复制按钮或滚动条；
- transcript 需要预留与按钮高度匹配的底部安全区；
- 显示未读更新数量，例如“回到最新 · 12”；
- 距离底部 80px 内自动隐藏；
- 点击后恢复跟随并滚到底部，行为保持 `prefers-reduced-motion` 兼容；
- 390px 宽度下按钮文本不得溢出，也不得遮挡发送按钮。

### 3.5 长回答目录与跳转

- 当回答至少包含 4 个标题或正文超过 3,000 字符时显示“目录”入口；
- 从 Markdown AST 生成稳定 heading ID，不从渲染后的 HTML 反向解析；
- 目录展示 H2/H3 层级，点击后只滚动 transcript 容器；
- 当前可见章节在目录中高亮，但不得造成频繁整页重渲染；
- 移动端目录使用轻量弹层或抽屉，并满足焦点锁定、Escape 和焦点归还；
- 标题文本重复时生成稳定且唯一的 ID；
- 刷新和流式完成后目录顺序必须与正文一致。

### 3.6 代码块体验

- 无明确语言时不突出显示 `text` 标签，可改为“Plain text”或隐藏；
- 复制操作使用图标按钮并提供 tooltip/accessible label；
- 复制成功状态短暂提示，不改变代码块尺寸；
- 超过 24 行或 4,000 字符的代码块默认折叠，展示行数；
- 展开/折叠不能改变 transcript 的跟随状态；
- 代码块只能自身横向滚动，不能产生页面级横向滚动；
- 后续可接入语法高亮，但本轮不得引入明显的首屏性能回退。

### 3.7 工具步数上限状态

- 不再把“Reached the maximum tool step limit for this turn.”仅当普通回答展示；
- 优先通过结构化事件或终态字段表达 `max_tool_steps`，禁止依赖脆弱的 UI 文本匹配；
- 显示专用状态卡，包含“本轮工具调用已达上限”、已用/上限数量和任务未完成说明；
- 提供“基于现有内容继续总结”操作；
- 该操作可预填并提交明确的继续指令，但必须遵守当前 turn/发送状态约束；
- 状态卡需要与 failed、cancelled、completed 明确区分；
- 刷新后该状态仍可从 snapshot 恢复。

### 3.8 可访问性与响应式

- Markdown 标题保持正确语义层级，不用纯样式 `div` 模拟标题；
- 工具分组和长代码使用原生 `details/summary` 或等价键盘语义；
- 目录、复制、展开和回到最新按钮均提供 accessible name；
- 状态变化使用简短独立 live region，不重复朗读整篇回答；
- 1280×720、768×1024、390×844 下无文本遮挡和页面横向溢出；
- 200% 浏览器缩放下主要操作仍可见、可聚焦、可执行；
- `prefers-reduced-motion` 下禁用非必要平滑滚动和展开动画。

## 4. 建议完成范围

- 回答内搜索和匹配项跳转；
- “全部折叠/全部展开”工具与代码块；
- 标题 permalink 和复制章节链接；
- 工具分组按成功/失败过滤；
- 回答顶部增加简短元信息，如耗时、工具次数和完成状态；
- Composer 草稿按 session 保存；
- 用户可选紧凑/舒适阅读密度。

## 5. 明确不在本任务范围

- 修改模型推理策略或自动生成新的回答摘要；
- 在线编辑代码、Diff 行级评论和 PR 工作流；
- 替换 WebSocket 核心事件顺序语义；
- 多 Agent、多工作区和团队协作；
- 引入完整 IDE、终端模拟器或文档编辑器；
- 为所有编程语言实现定制语法分析。

## 7. 验收用例

### AC-READ-01 Markdown 语义

输入包含 H1/H2/H3、粗体、行内代码、列表、表格、引用、分隔线和 fenced code 的回答：

- 页面中不再裸露作为语法的 `###`、`**` 和围栏反引号；
- 标题具有正确 heading role；
- 表格和代码在窄屏下只滚动自身；
- `<script>`、事件属性和 `javascript:` 链接不执行。

### AC-READ-02 工具聚合

同一 turn 连续执行 13 次 `read_file`：

- 默认只显示 1 个工具分组；
- 摘要数量、成功数和累计耗时正确；
- 展开后可看到全部 13 项，顺序与 seq 一致；
- 插入 Assistant 消息后，前后工具不得错误合并。

### AC-READ-03 回到最新

在 1280×720 和 390×844 下向上滚动长回答：

- 按钮出现在 transcript 右下安全区；
- 按钮与正文、代码工具栏、composer 的重叠面积为 0；
- 未读数量随新 delta 增长；
- 点击后底部距离不超过 2px，按钮隐藏。

### AC-READ-04 长回答目录

加载包含 10 个标题、40,000 字符的回答：

- 自动出现目录入口；
- 点击第 8 个标题后只改变 transcript.scrollTop，`window.scrollY` 保持 0；
- 重复标题具有不同 ID；
- 移动端关闭目录后焦点返回入口按钮。

### AC-READ-05 代码块

- 10 行代码默认展开，30 行代码默认折叠；
- 复制内容与原始代码完全一致；
- 展开长代码不会错误触发“回到最新”或抢回用户滚动位置；
- 300 字符长行不撑宽 document。

### AC-READ-06 工具上限

模拟达到最大工具步数：

- 显示专用未完成状态，而不是普通最终回答；
- 明确展示使用数量和继续操作；
- 点击继续后生成合法的新 turn，不重复旧工具事件；
- 刷新后状态卡仍然存在。

### AC-READ-07 视觉与无障碍

- 桌面、平板、移动端和 200% 缩放无重叠；
- Tab 可以访问所有交互控件，焦点顺序与视觉顺序一致；
- 浏览器控制台无 React key warning 和运行时错误；
- axe 或等价基础扫描无 serious/critical 问题。

## 8. 测试要求

### 前端单元/组件测试

- Markdown 全语法样例和脚本注入回归；
- 流式未闭合 Markdown 围栏；
- 工具分组边界、失败状态、耗时和 seq 顺序；
- 回到最新位置、未读计数和 reduced-motion；
- heading ID 去重、目录生成和容器内跳转；
- 长代码折叠与复制；
- 工具上限状态恢复和继续操作；
- 移动目录/抽屉的焦点锁定和 Escape。

### 后端/协议测试

- 工具上限结构化终态及 snapshot 恢复；
- 旧 session 不含新字段时保持向后兼容；
- reconnect 后状态和工具事件不重复；
- Secret 脱敏继续覆盖 Markdown、工具输入和错误内容。

### 浏览器测试

- 1280×720 长回答排版和按钮零遮挡；
- 390×844 表格、代码和目录；
- 13 个连续工具调用聚合/展开；
- 40,000 字符、10 个标题的目录跳转；
- 达到工具步数上限后的继续流程；
- 刷新和 WebSocket 重连恢复；
- 控制台 warning/error 检查。

## 9. 性能预算

- 40,000 字符回答首次渲染不得出现超过 200ms 的持续主线程阻塞；
- 13 个同类工具默认只创建 1 个可见分组容器，明细仅在展开后完整渲染；
- 目录当前章节更新应节流，不得在每个 scroll event 上同步重渲染整棵 Markdown；
- 生产 JavaScript gzip 建议保持在 85KB 以内；超过时必须在实现报告中说明原因；
- Markdown 渲染结果应按消息内容缓存，非内容状态更新不得重复解析全文；
- 浏览器性能验收时 document 继续保持视口高度，不得恢复整页滚动。

## 10. 代码注释与文档要求

- 注释解释 Markdown 安全边界、工具分组边界和 heading ID 稳定性；
- 不为显而易见的 JSX/CSS 添加重复性注释；
- 新增第三方 Markdown 依赖时记录选择原因、安全策略和 bundle 影响；
- 实现报告记录前后对比截图、DOM 数量、bundle 大小和浏览器指标；
- 测试报告逐项映射 AC-READ-01 至 AC-READ-07；
- Git commit 使用 `feat:` / `fix:` / `perf:` / `test:` / `docs:`，正文列出验证命令与结果。

## 11. Definition of Done

- AC-READ-01 至 AC-READ-07 全部通过；
- 截图中的原始 Markdown 标记问题消失；
- 连续工具调用不再淹没回答正文；
- “回到最新”在所有目标视口都不遮挡内容；
- 长回答具有可用目录和稳定标题跳转；
- 工具步数上限具有明确、可恢复、可继续的状态；
- 完整 Python、前端测试和生产构建通过；
- 浏览器控制台无错误，`git diff --check` 通过；
- 完成报告和测试报告同步写入 `docs/newTask/`。
